# Phase 2 详细方案：用户体验 + 数据可视化 + 智能对话

## 当前短板分析

| 模块 | 现状 | 问题 |
|------|------|------|
| Chat 对话 | 单轮 stream，无上下文窗口管理 | 模型看不到历史消息，无法多轮对话 |
| 对话记忆 | 消息只存 DB，无缓存无搜索 | 历史对话找不到，无法回溯 |
| 用户控制台 | 4 个数字卡片 + 热力图 + 表格 | 数据维度单一，无趋势图、无对比 |
| 模型网络页 | 模型卡片只有名称/节点数/价格 | 无性能数据、无使用排名、无质量评分 |
| Admin 面板 | 4 个数字 + top10 表格 | 无实时监控、无趋势、无告警 |

## 模块 A：智能多轮对话 + 对话缓存

### A1. 多轮上下文窗口管理

**问题**：当前 `/v1/chat/conversations/{id}/stream` POST 只发最后一条 `content`，provider 看不到历史。

**方案**：

```
用户发送消息
      │
      ▼
从 DB 加载该 conversation 的历史 messages
      │
      ▼
上下文窗口裁剪（按 token 预估）
  ├─ 保留 system prompt（如有）
  ├─ 从最新往回取 messages
  ├─ 估算 token 数 ≈ content.length / 3.5
  ├─ 超过模型限制的 80% 时停止
  └─ 至少保留最近 2 轮（4 条消息）
      │
      ▼
拼装完整 messages[] 发给 provider
      │
      ▼
流式返回 + settlement 结算（按实际 usage）
```

**后端改动**：
- `main.ts` conversation stream handler：加载历史 messages，裁剪后拼装发给 provider executor
- 新增 model context window 配置（per-model max tokens 限制）
- 新增 `system_prompt` 字段到 conversation 表

**前端改动**：
- Chat 设置面板：每个对话可配置 system prompt
- 显示当前上下文 token 估算值（输入框上方小字）

### A2. 对话缓存与离线加载

**问题**：每次切换对话都重新从服务器加载，慢且无离线支持。

**方案**：

```
┌───────────────────────────────────────┐
│           IndexedDB (本地)             │
│                                       │
│  conversations/  ← 对话列表缓存       │
│  messages/{convId}/  ← 消息缓存       │
│  metadata/lastSync  ← 同步时间戳      │
└───────────────────┬───────────────────┘
                    │
        ┌───────────▼───────────┐
        │   Cache-First 策略     │
        │                       │
        │  1. 先读本地 IndexedDB │
        │  2. 异步拉取 server   │
        │  3. 增量合并新消息     │
        │  4. 写回本地缓存       │
        └───────────────────────┘
```

**实现**：
- 使用 `idb-keyval` 或原生 IndexedDB
- 每个 conversation 存 `{messages[], updatedAt}`
- 切换对话时：立即显示缓存 → 后台 sync → 增量更新
- 流式消息完成后同步写入本地

### A3. 历史对话搜索

**问题**：对话多了找不到之前的内容。

**方案**：

```
┌─────────────────────────────────┐
│  🔍 搜索对话                     │  ← 搜索框在对话侧栏顶部
│  [输入关键词...]                 │
├─────────────────────────────────┤
│  搜索结果：                      │
│  📄 "MiniMax 性能测试"           │  ← 匹配对话标题
│     ...关于 M2.7 的延迟测试...    │  ← 匹配的消息片段高亮
│  📄 "API 接入问题"               │
│     ...provider key 验证失败...   │
└─────────────────────────────────┘
```

**后端**：
- `GET /v1/chat/conversations/search?q={keyword}&limit=20`
- SQL：`WHERE (title ILIKE '%keyword%' OR EXISTS (SELECT 1 FROM chat_messages WHERE content ILIKE '%keyword%' AND conversation_id = c.id))`
- 返回匹配的 conversation + 第一条匹配 message 片段（上下文 50 字）

**前端**：
- 侧栏搜索框，debounce 300ms
- 搜索结果列表显示标题 + 匹配片段（关键词高亮）
- 点击跳转到对应对话并滚动到匹配消息
